Bug Summary

Problem: GET /shipments Returns Null Names for Newly Created Shipments

When shipments are created via POST /shipments, the GET /shipments list endpoint returns null for all denormalized name fields (site_name, customer_name, vendor_name). The IDs are correct and reference valid records, but the names are not populated.

Example Response (Before Fix)

{
  "id": "8d297e4c-16e5-4c26-b215-9b9add01209f",
  "site_id": "site-001",
  // These are all null!
  "site_name": null,
  "customer_id": "cust-002",
  "customer_name": null,
  "vendor_id": "vend-006",
  "vendor_name": null
}

Fix Implementation

Implemented

Solution: JOIN on Read (Recommended Approach)

The list_shipments() function now uses LEFT JOINs to fetch entity names directly from their source tables. This ensures names are always fresh and consistent, even if the original entity is renamed.

Updated SQL Query

SELECT
  s.*,
  st.name AS site_name,
  c.name AS customer_name,
  v.vendor_name AS vendor_name,
  wt.name AS waste_type,
  wt.waste_category AS waste_category,
  t.transporter_name AS carrier_name,
  rf.facility_name AS facility_name
FROM shipments s
LEFT JOIN sites st ON s.site_id = st.id
LEFT JOIN customers c ON s.customer_id = c.id
LEFT JOIN approved_vendors v ON s.vendor_id = v.id
LEFT JOIN waste_types wt ON s.waste_type_id = wt.id
LEFT JOIN transporters t ON s.transporter_id = t.id
LEFT JOIN receiving_facilities rf ON s.receiving_facility_id = rf.id
WHERE {where_clause}
ORDER BY s.created_at DESC
LIMIT %s OFFSET %s

Python Implementation

def list_shipments(event):
    """List shipments with pagination, filtering, and JOINed names"""
    query_params = event.get('queryStringParameters') or {}

    # ... pagination and filter setup ...

    with conn.cursor() as cursor:
        # JOIN query to get names from related tables
        cursor.execute(f'''
            SELECT
                s.*,
                st.name AS site_name,
                c.name AS customer_name,
                v.vendor_name AS vendor_name,
                wt.name AS waste_type,
                wt.waste_category AS waste_category,
                t.transporter_name AS carrier_name,
                rf.facility_name AS facility_name
            FROM shipments s
            LEFT JOIN sites st ON s.site_id = st.id
            LEFT JOIN customers c ON s.customer_id = c.id
            LEFT JOIN approved_vendors v ON s.vendor_id = v.id
            LEFT JOIN waste_types wt ON s.waste_type_id = wt.id
            LEFT JOIN transporters t ON s.transporter_id = t.id
            LEFT JOIN receiving_facilities rf ON s.receiving_facility_id = rf.id
            WHERE {where_clause}
            ORDER BY s.created_at DESC
            LIMIT %s OFFSET %s
        ''', params + [limit, offset])

        shipments = cursor.fetchall()
        return create_response(200, {'shipments': shipments, ...})

Answer: Batch Shipment Creation Endpoint

EXISTS

Question from Mcjethro Kalu: "We do not have endpoint for creating new shipments in batch"

Answer: The batch creation endpoint already exists. Use POST /shipments with a shipments array in the request body. The backend automatically detects batch mode and processes all shipments in a single transaction.

Batch Creation Request Format

{
  "shipments": [
    {
      "site_id": "site-001",
      "customer_id": "cust-002",
      "vendor_id": "vend-006",
      "waste_type_id": "wt-003",
      "shipment_date": "2026-03-19",
      "quantity": 500,
      "quantity_unit": "kg"
    },
    {
      "site_id": "site-002",
      "customer_id": "cust-002",
      "vendor_id": "vend-007",
      "waste_type_id": "wt-005",
      "shipment_date": "2026-03-19",
      "quantity": 1200,
      "quantity_unit": "kg"
    }
  ]
}

Response

{
  "inserted": 2,
  "ids": ["uuid-1", "uuid-2"],
  "errors": null,
  "message": "2 shipments created"
}

Batch Creation Features
  • Auto-resolves entity names from IDs (site_name, customer_name, vendor_name, waste_type)
  • Validates all foreign keys before insert
  • Removes empty FK fields to avoid constraint errors
  • Creates audit log entry for bulk operation
  • Returns partial success with error details per row
